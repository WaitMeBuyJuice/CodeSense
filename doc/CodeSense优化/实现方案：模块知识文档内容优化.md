## 一、决策汇总

| 决策点           | 结论                                                                                |
| ------------- | --------------------------------------------------------------------------------- |
| 存储结构          | nested：`.codesense/modules/<module>.md` + `.codesense/modules/<module>/<file>.md` |
| `__init__.py` | 不生成独立子模块文档，导出信息合并到模块总览的「文件清单」章节                                                   |
| 单文件模块         | 跳过 P0.4，只生成模块总览                                                                   |
| 缓存粒度          | 模块总览 hash 与子模块 hash 独立，互不连带失效                                                     |
| 工具接口          | 新增 3 个：`explore_submodule` / `get_submodule_prompt` / `save_submodule_summary`    |
| 模块总览章节        | 5 个：一句话定位、架构简析、文件清单、上下游关系、实现约束清单                                                  |
| 子模块文档章节       | 4 个：文件概述、对外接口、跨模块依赖、典型调用链                                                         |
| 已剔除           | 关键常量字面量（生成慢、失效频繁、价值散落在子模块层）                                                       |

## 二、目标产出结构

```
.codesense/
└── modules/
    ├── .hashes.json                ← 模块总览 hash（保持现状，key = module safe_key）
    ├── <module>.md                 ← 模块总览（5 章节）
    └── <module>/                   ← 子模块目录（仅多文件模块创建）
        ├── .hashes.json            ← 子模块 hash 表（key = 文件 safe_key）
        └── <file>.md               ← 单文件子模块文档（4 章节）
```

例：`cache/` 模块产出

```
modules/
├── cache.md                       ← 模块总览（含 __init__.py 导出信息）
└── cache/
    ├── .hashes.json
    └── cache.md                   ← 对应 src/codesense_v1/cache/cache.py
```

例：`errors`（单文件模块 errors.py）产出

```
modules/
└── errors.md                       ← 仅模块总览
```

## 三、模块总览章节（P0.3）

| 章节     | 数据来源                                                          | 由谁填                                                                      |
| ------ | ------------------------------------------------------------- | ------------------------------------------------------------------------ |
| 一句话定位  | `modules_index.json` description                              | LLM                                                                      |
| 架构简析   | `nodes` 表（各文件符号名/kind）+ `__init__.py` 源码 + Agent 自主 read_file | LLM                                                                      |
| 文件清单   | `files` 表内模块目录文件 + 各文件 docstring + `__init__.py` 导出           | LLM                                                                      |
| 上下游关系  | `module_dependencies()` 跨模块 imports 边                         | LLM（数据全 `extracted`，无需推断置信度；但保留字段以便未来扩展）                                 |
| 实现约束清单 | Agent 自主读模块内源文件                                               | LLM；prompt 显式注明「只写模块内部边界行为，跨模块架构禁忌在 project_map 04_constraints 已有，此处不重复」 |

## 四、子模块文档章节（P0.4）

| 章节    | 数据来源                                                           | 由谁填                    |
| ----- | -------------------------------------------------------------- | ---------------------- |
| 文件概述  | `nodes` 表（本文件符号集）+ 文件 docstring                                | LLM                    |
| 对外接口  | `nodes`（kind=function/class/method 且非下划线开头）+ `nodes.signature` | LLM（程序提供签名，LLM 补一句话说明） |
| 跨模块依赖 | `edges` 表 `imports`/`calls` 以本文件为 source 或 target              | 程序整理边数据，LLM 输出表格       |
| 典型调用链 | `calls` 边多跳追踪（以本文件公开符号为起点）                                     | LLM                    |

「单文件模块」判定规则：模块目录下除 `__init__.py` 外只有 1 个或 0 个 `.py` 文件 → 跳过子模块文档生成。

## 五、缓存与 Hash 设计

### 5.1 模块总览 Hash（保持现状）

- 函数：`summarizer._compute_module_hash(entry, db)` 不动
- 范围：模块目录全部 `.py` 文件的 nodes 指纹 + 文件列表
- 存储：`.codesense/modules/.hashes.json`

### 5.2 子模块 Hash（新增）

- 新函数：`summarizer._compute_submodule_hash(file_path, db) -> str`
- 范围：
  - 该文件在 `nodes` 表的全部节点指纹（`name:kind:signature` 排序拼接）
  - 该文件的 `edges` 中以本文件为 source 的 `imports` 边集合（目标文件路径）
  - 该文件的 `edges` 中 `calls` 边集合（source 在本文件、target 任意）
- 存储：`.codesense/modules/<module>/.hashes.json`
- 失效独立：改 A.py → 只失效 `modules/<module>/A.md` + 整个模块总览（总览 hash 包含全模块数据，会一并失效）

## 六、代码改动清单

### 6.1 cache 层（`src/codesense_v1/cache/cache.py`）

新增函数：

| 函数                      | 签名                                                                       | 作用                             |
| ----------------------- | ------------------------------------------------------------------------ | ------------------------------ |
| `read_submodule`        | `(codesense_dir, module_key, file_key) -> str \| None`                   | 读 `modules/<module>/<file>.md` |
| `write_submodule`       | `(codesense_dir, module_key, file_key, content, submodule_hash) -> None` | 写文档 + 更新对应 `.hashes.json`      |
| `read_submodule_hashes` | `(codesense_dir, module_key) -> dict[str, str]`                          | 读子模块 hash 表                    |
| `write_submodule_hash`  | `(codesense_dir, module_key, file_key, submodule_hash) -> None`          | 单独写一个 hash 条目                  |
| `submodule_path`        | `(codesense_dir, module_key, file_key) -> Path`                          | 内部路径计算                         |

`safe_key` 函数复用：模块名和文件名均通过它转换为文件系统安全 key（文件名形如 `cache_py`，避免点号污染目录）。

### 6.2 summarizer 层（`src/codesense_v1/summarizer/summarizer.py`）

新增：

| 函数                                                                                           | 作用                                               |
| -------------------------------------------------------------------------------------------- | ------------------------------------------------ |
| `_compute_submodule_hash(file_path, db)`                                                     | 见 5.2                                            |
| `_build_submodule_prompt(module_entry, file_path, file_nodes, file_edges, ref_docs_section)` | 构造子模块 LLM prompt                                 |
| `save_submodule_summary(project_root, module_name, file_path, summary)`                      | 校验入参（文件确属该模块）→ 计算 hash → 调 cache.write_submodule |
| `_is_single_file_module(entry)`                                                              | 判断模块是否单文件，子模块工具用此跳过                              |

修改：

- `_build_module_prompt` 调整 prompt 结构以匹配新 5 章节，显式说明：
  - 「架构简析」要求基于文件分层
  - 「文件清单」要求每文件一句话简介，且 `__init__.py` 导出要单列
  - 「实现约束清单」加入与 04_constraints 的分工说明

### 6.3 data 层（`src/codesense_v1/data/`）

可能新增一两个文件级查询辅助：

| 函数（位置）                                       | 作用                                       |
| -------------------------------------------- | ---------------------------------------- |
| `modules.py:file_symbols_for(db, file_path)` | 取单文件全部 nodes（function/class/method）      |
| `modules.py:file_edges_for(db, file_path)`   | 取以该文件为 source 或 target 的 imports/calls 边 |

如果现有 `module_dependencies` / `iter_nodes` / `iter_edges` 已能简单拼出，则不新增、直接在 summarizer 里拼。

### 6.4 tools 层（新增 3 个 MCP 工具）

文件结构：

```
src/codesense_v1/tools/
├── explore_submodule.py          ← 新增
├── get_submodule_prompt.py       ← 新增
└── save_submodule_summary.py     ← 新增
```

工具签名与行为：

#### `explore_submodule(module_name: str, file_path: str) -> str`

- 行为：与 `explore_module` 完全对称
  1. 校验 codegraph.db / modules_index 存在
  2. 解析 module_name 对应 entry
  3. 单文件模块 → 返回提示「该模块为单文件模块，请使用 `explore_module`」
  4. 校验 file_path 属于该模块
  5. 计算 `_compute_submodule_hash`
  6. 读 `modules/<module>/.hashes.json` 对比
  7. 命中 → 返回 .md；未命中 → 返回引导（让 Agent 调 `get_submodule_prompt` → 生成 → `save_submodule_summary`）

#### `get_submodule_prompt(module_name: str, file_path: str) -> str`

- 行为：调 `_build_submodule_prompt` 返回 prompt 文本

#### `save_submodule_summary(module_name: str, file_path: str, summary: str) -> str`

- 行为：调 `summarizer.save_submodule_summary`，返回写入路径

参数命名说明：`file_path` 接受相对项目根的 POSIX 路径（与 modules_index 中 `files` 字段一致），便于 Agent 直接从 `explore_module` 输出的文件清单复制。

### 6.5 server 注册

`src/codesense_v1/server.py`（或工具注册中心）注册新增 3 个工具，与现有 11 个工具并列。最终 14 个 MCP 工具。

## 七、生成流程

### 7.1 Agent 主路径

```
用户问"<module> 模块做什么"
  ↓
Agent 调 explore_module(<module>)
  ↓ 缓存命中 → 直接返回模块总览
  ↓ 缓存未命中
Agent 调 get_module_prompt(<module>) → LLM 生成 → save_module_summary
  ↓
Agent 再次调 explore_module 取最新内容
```

```
用户问"<module>/<file> 的对外接口"
  ↓
Agent 调 explore_submodule(<module>, <file>)
  ↓ 单文件模块 → 提示用 explore_module
  ↓ 缓存命中 → 返回子模块文档
  ↓ 缓存未命中
Agent 调 get_submodule_prompt → LLM 生成 → save_submodule_summary
  ↓
Agent 再次调 explore_submodule 取最新内容
```

### 7.2 与现有工具的关系

- `project_map` 不变；模块总览的「上下游关系」与 07_dependencies 数据同源但展示角度不同（模块总览站在某一模块视角，07 是全局表）。
- `explore_module` 行为不变，只是它返回的内容章节更丰富。
- 7.2 旧缓存兼容：`.codesense/modules/<module>.md` 在格式升级后旧内容仍可读；但 hash 算法未变 → 不会自动失效。**实施前需要让用户手动删除 `.codesense/modules/` 目录一次**，触发全量重生成。

## 十、风险与回退

| 风险                                    | 缓解                                                     |
| ------------------------------------- | ------------------------------------------------------ |
| Prompt 复杂度上升导致 LLM 输出格式不稳             | 子模块 prompt 比模块总览简单，可逐项 fewshot；模块总览先小范围灰度（手动测 2-3 个模块） |
| 子模块文档数量暴增（10 模块 × 平均 5 文件 = 50 份 .md） | 全部 Lazy 生成，没人查询就不生成；hash 独立失效成本可控                      |
| 旧缓存与新格式冲突                             | 实施时通过删除 `.codesense/modules/` 解决，无需迁移脚本                |

如需回退：删除 3 个新工具文件 + server 取消注册 + 还原 `_build_module_prompt`，cache 层新增函数留着无害。
