# Vibe Coding 步骤

## 全局规则

所有步骤生成的代码必须遵守以下规则：

- 使用 `pytest` 编写完整单元测试，覆盖正常路径和异常路径
- 通过 `mypy` 静态类型检查（strict 模式）
- 通过 `ruff` 代码风格检查
- 任何不明确的地方，禁止猜测，必须向我提问

---

## 步骤 0：技术选型

### Agent输入提示词：

```
目标:
确定项目的技术栈和工程规范，作为后续所有步骤的基准

输入:
我对项目的描述：XXX

输出:
doc/stack.md

内容要求:
- 编程语言及版本
- 核心框架和依赖库（含版本约束）
- 项目目录结构约定
- 编码规范（命名、注释、类型注解要求）
- 测试规范（测试框架、覆盖率要求）
- 构建与运行方式

约束:
1.不要猜测我的意图，任何不明确的地方都必须向我提问。
2.技术选型必须经过我确认后才能输出文档。
```

## 步骤 1：确定需求

### Agent输入提示词：

```
目标:
通过对话澄清需求，生成需求文档

输入:
我的目标描述：XXX

输出:
doc/requirement.md

内容要求:
- 项目背景与目标
- 用户角色与使用场景
- 功能需求列表（按优先级排序）
- 非功能需求（性能、安全、兼容性等）
- 明确的输入/输出定义

约束:
1.使用提问的方式帮助我逐条确认需求。
2.不要猜测我的意图，任何不明确的地方都必须向我提问。
3.所有需求条目必须可验证（能写成测试用例）。
```

## 步骤 3：详细设计

### Agent输入提示词：

```
目标:
为每个模块生成详细设计文档

输入:
概要设计 doc/design/overview.md
需求文档 doc/requirement.md

输出:
doc/design/<module-name>.md（每个模块对应一个）

内容要求（每份文档包含）:
- 模块功能说明
- 对外暴露的接口签名（函数/类/API，含类型注解）
- 核心数据结构定义
- 错误码与异常处理规范
- 关键算法或业务逻辑说明
- 与其他模块的交互契约

约束:
1.接口签名必须使用目标语言的完整类型注解。
2.不要猜测我的意图，任何不明确的地方都必须向我提问。
3.所有模块详细设计完成后需汇总确认，再进入下一步。
```

## 步骤 4：划分任务

### Agent输入提示词：

```
目标:
将每个模块拆分为最小可执行任务

输入:
需求文档 doc/requirement.md
概要设计 doc/design/overview.md
详细设计 doc/design/<module-name>.md

输出:
doc/tasks/<module-name>.md（每个模块对应一个）
doc/tasks/progress.md（总体进度）

任务粒度标准:
- 单个任务对应不超过 1 个文件的改动
- 每个任务必须包含：任务描述、输入、输出、验收标准（含测试用例描述）
- 标注任务间依赖关系（"依赖任务 X 完成后才能开始"）

输出格式（<module-name>.md）:
- [ ] 任务ID: 任务描述
  - 输入: ...
  - 输出: ...
  - 验收标准: ...
  - 依赖: 任务ID（无则填"无"）

输出格式（progress.md）:
- [ ] <module-name>（包含任务数量和完成数量）

约束:
1.若没有doc/design/<module-name>.md，则询问我是否要先进行详细设计，还是直接基于doc/design/overview.md拆任务
1.任务粒度不符合标准时必须继续拆分。
2.不要猜测我的意图，任何不明确的地方都必须向我提问。
3.<module-name>.md中用check list表示子任务是否完成。
4.progress.md中用check list表示模块是否已经完成。
```

## 步骤 5：生成执行提示词

### Agent输入提示词：

```
目标:
为每个任务生成 Vibe Coding 用的独立 Prompt

输入:
技术选型 doc/stack.md
需求文档 doc/requirement.md
详细设计 doc/design/<module-name>.md
任务列表 doc/tasks/<module-name>.md

输出:
doc/prompts/<task-id>.md（每个任务对应一个）
doc/prompts/index.md（所有 prompt 的索引，含执行顺序）

每个 prompt 文件内容要求:
- 任务背景（引用需求和设计中的相关段落）
- 明确的实现目标
- 接口契约（输入/输出/类型）
- 需要实现的文件路径
- 测试文件路径及测试用例要求
- 验收标准（通过 pytest / mypy / ruff）

Agent 执行模式:
主 Agent 职责：
  - 持有并维护 doc/tasks/progress.md
  - 按依赖顺序分配任务给子 Agent
  - 子 Agent 完成后验证测试结果，更新 progress.md
  - 发现阻塞或失败时中止并报告，不自动绕过

子 Agent 职责：
  - 领取单个任务的 prompt
  - 实现代码 + 编写测试
  - 运行 pytest / mypy / ruff，全部通过后返回结果
  - 不得修改任务范围外的文件

约束:
1.生成 prompt 过程中，如有任何不明确的地方都必须向我提问。
2.prompt 必须自包含，子 Agent 无需读取其他文档即可执行。
```

## 步骤 6：执行与验证

### Agent输入提示词：

```
目标:
按任务顺序执行代码实现，确保每个任务独立可验证

执行流程:
1. 主 Agent 读取 doc/tasks/progress.md，找到第一个未完成且依赖已满足的任务
2. 加载对应的 doc/prompts/<task-id>.md，启动子 Agent
3. 子 Agent 完成实现后，必须执行：
   - pytest（全部通过）
   - mypy --strict（零错误）
   - ruff check（零警告）
4. 三项全部通过：
   - 勾选 doc/tasks/<module-name>.md 中对应条目
   - 更新 doc/tasks/progress.md
5. 任意一项失败：
   - 将失败信息写入 doc/tasks/<module-name>.md 的「缺陷记录」区
   - 子 Agent 修复后重新验证，不得跳过
6. 重复步骤 1，直到所有任务完成

约束:
1.禁止在测试未通过的情况下勾选任务完成。
2.禁止修改任务范围外的文件（需主 Agent 授权）。
3.遇到阻塞问题必须向我报告，不得自行决策绕过。
```

## 步骤 6（单Agent版）：执行与验证

### Agent输入提示词：

```
目标:
按任务顺序执行代码实现，确保每个任务独立可验证

输入:
doc/tasks/progress.md（任务总进度）
doc/tasks/<module-name>.md（各模块任务列表）
doc/prompts/<task-id>.md（各任务的自包含 prompt）

执行流程:
1. 读取 doc/tasks/progress.md，找到第一个未完成且依赖已满足的任务 ID
2. 读取对应的 doc/prompts/<task-id>.md，严格按照其中的"实现目标"与"内容契约"编写代码
3. 仅创建/修改该 prompt 中"需要实现的文件"所列出的文件，不触碰任何其他文件
4. 实现完成后，依次执行以下验证命令：
   a. uv run ruff check <file>（零警告）
   b. uv run mypy --strict <file>（零错误）
   c. uv run pytest（全量，全部通过）
5. 三项全部通过：
   - 在 doc/tasks/<module-name>.md 中将对应条目从 [ ] 改为 [x]
   - 更新 doc/tasks/progress.md 中的完成数量；若模块全部子任务完成则勾选模块条目
6. 任意一项失败：
   - 分析失败原因，在 doc/tasks/<module-name>.md 末尾的「缺陷记录」区追加失败信息与修复说明
   - 修复代码后回到步骤 4 重新验证，不得跳过验证直接进入下一任务
7. 回到步骤 1，重复直到 doc/tasks/progress.md 中所有任务均完成

约束:
1. 每次只处理一个任务，完成验证后才进入下一个。
2. 严格只改 prompt 中声明的"需要实现的文件"，不得修改其他任何文件。
3. 禁止在三项验证未全部通过的情况下勾选任务完成。
4. 验证命令必须实际执行并读取输出，不得凭假设认为"应该通过"。
5. 如果修复同一任务超过 3 次仍失败，停止并向我报告具体失败信息，等待指令。
6. 遇到 prompt 内容与实际代码/SDK 有出入时，在任务允许范围内做最小适配；若必须改动范围外文件，停止并报告。
```

## 步骤 7：复盘与重构（暂时不需要）

### Agent输入提示词：

```
目标:
完成所有任务后，对代码库进行整体复盘和清理

输入:
全部源码
doc/requirement.md
doc/design/overview.md

检查项:
- [ ] 重复代码检测：提取公共函数/类，消除 DRY 违规
- [ ] 死代码清理：删除未被调用的函数、变量、导入
- [ ] 类型注解完整性：确保所有公开接口有类型注解
- [ ] 文档与代码对齐：接口签名与设计文档一致
- [ ] 测试覆盖率：关键路径覆盖率 ≥ 80%
- [ ] 依赖清理：移除未使用的第三方库

输出:
- 重构后的代码（原地修改）
- doc/refactor-log.md（记录所有改动及理由）

约束:
1.重构不得改变任何对外接口的行为（有测试保障）。
2.每次重构改动后必须重新运行全量 pytest / mypy / ruff。
3.有疑问的改动必须向我确认，不得擅自删除代码。
```
