# 任务 2：explore_module 调用流程分析（CodeSense_V1，复杂，纯问答）

## 一、实验内容

**任务类型**：跨模块流程分析问答（无代码改动，测调用链完整性 + 关键节点命中率）

**任务描述**：
> 我想优化 `explore_module` 工具的响应速度。请帮我分析：从 MCP Server 收到 `explore_module` 调用请求，到最终返回结果给 Agent，整个过程经过了哪些模块、各自做了什么、有哪些可能的性能瓶颈点（按调用顺序列出）。

**涉及项目**：`E:\Python_Project\CodeSense_V1`

**为什么选这个任务**：
- 不在提示词出现"用 CodeSense"等暗示
- "经过了哪些模块、各自做了什么" 强烈命中 `project_map` 和 `explore_module` description
- 调用链跨 6 个模块（server → registry → tools → summarizer → cache/llm/data），需要完整理解
- **反假饱和关键观察**：组3/4 会不会只看 project_map 就给出概略，不深入到 cache/llm 层
- 纯问答无代码改动，**实验后无需重置代码库**

**Ground Truth（基于实际代码核实）**：

完整调用链（按顺序）：

| # | 模块 | 文件 | 做的事 |
|---|-----|------|-------|
| 1 | server | `src/codesense_v1/server/server.py` | 接收 MCP `tools/call` 请求，分发到 `_call_tool` → `registry.dispatch` |
| 2 | registry | `src/codesense_v1/registry/registry.py` | JSON Schema 校验输入参数，找到 `explore_module` handler，调用之 |
| 3 | tools | `src/codesense_v1/tools/explore_module.py` | 取 `CODESENSE_PROJECT_ROOT`、校验参数、调 `summarizer.module_summary` |
| 4 | summarizer | `src/codesense_v1/summarizer/summarizer.py` | 读 modules_index.json、定位模块、构建 prompt、调 LLM、写缓存 |
| 5 | cache | `src/codesense_v1/cache/cache.py` | 读/写 modules_index.json、modules/*.json、project_map.md |
| 6 | data | `src/codesense_v1/data/` | CodeGraphDB 查询：iter_files、iter_nodes、aggregate.directory_dependencies |
| 7 | llm | `src/codesense_v1/llm/llm.py` | 调用 OpenAI 兼容 API 生成模块摘要 |

**Ground Truth 关键概念清单**（共 12 项）：
1. `server.py` 中 `_call_tool` 入口
2. `registry.dispatch` 做参数校验
3. `tools/explore_module.py` 校验 `module_name`
4. `summarizer.module_summary` 是核心协调器
5. 读取 `.codesense/modules_index.json` 查找模块
6. 缓存命中检查（hash 一致直接返回）
7. CodeGraph 数据库查询（`data/` 模块）
8. 构建 LLM prompt（含目录依赖、符号列表）
9. **LLM 调用是主要性能瓶颈**
10. 缓存写入 `.codesense/modules/<key>.json`
11. 返回 Markdown 字符串
12. 提到性能瓶颈：LLM 调用延迟、DB 全量遍历、JSON 解析

**验收**：无自动验收，人工对照计算命中率。

**实验次数**：每组 3 次（关键任务）
