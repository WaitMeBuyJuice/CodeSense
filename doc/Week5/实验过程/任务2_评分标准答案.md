## 完整标准调用链（共 7 层）

```
① server.py   _call_tool()
   ↓ registry.dispatch("explore_module", {module_name: "..."})

② registry.py  dispatch()
   jsonschema 校验 module_name → 从 _REGISTRY 找 handler → 调用 explore_module(module_name)

③ tools/explore_module.py  explore_module()
   strip/validate module_name → 读取 CODESENSE_PROJECT_ROOT → 调 summarizer.module_summary()

④ summarizer.py  module_summary()
   ├─ cache.db_hash(.codegraph/codegraph.db)     ← 仅 CODESENSE_CACHE_AUTO_EXPIRE=true 时执行
   ├─ cache.read_module(codesense_dir, mkey)      ← 缓存命中检查
   │     命中 → 直接返回（跳过 ⑤⑥⑦，极快）
   │     未命中 → 继续
   ├─ cache.read_modules_index()                  ← 查 modules_index.json，定位模块文件列表
   └─ 开 CodeGraphDB → 查数据（⑤）→ 调 LLM（⑦）→ 写缓存（⑥）

⑤ data/ （db.py + modules.py + aggregate.py）
   list_modules(db)           ← 文件列表
   module_dependencies(db)    ← 文件间 import/call 边
   directory_dependencies()   ← 聚合到目录级
   db.iter_nodes(kinds=(function/class/method))  ← 全量遍历节点

⑥ cache.py
   读: read_module / read_modules_index（JSON 解析）
   写: write_module（JSON 序列化 + 写磁盘）

⑦ llm.py  call_llm()
   调用 OpenAI 兼容 API，传入含文件列表/符号签名/上下游依赖的 prompt
   → 接收返回的模块 Markdown 描述
```

---

## 性能瓶颈标准答案

| 瓶颈点 | 严重程度 | 说明 |
|--------|---------|------|
| **LLM 调用（⑦）** | ★★★★★ 最主要 | 网络 I/O + 推理延迟，通常 3-10 秒，无法规避 |
| **DB 全量节点遍历（⑤）** | ★★★ | `iter_nodes` 扫描所有 function/class/method，大项目 O(N)，当前项目较小影响有限 |
| **db_hash 计算** | ★★ | 对 codegraph.db 做完整 SHA-256（文件可达几十 MB），**仅在 CODESENSE_CACHE_AUTO_EXPIRE=true 时触发** |
| **JSON 读写（⑥）** | ★ | modules_index.json 读写，量小，基本可忽略 |
| **缓存完全命中时** | 无瓶颈 | 读缓存文件直接返回，几毫秒 |

> ⚠️ **重要**：当前默认模式（`CODESENSE_CACHE_AUTO_EXPIRE` 未设为 `true`）下，`module_summary` **不做 db_hash 检查**，直接调用 `cache.read_module()`（见 `summarizer.py:206-208`）。因此缓存命中时，db_hash 完全不执行。只有环境变量开启或缓存 miss 时才涉及 db_hash。

## 6 项评分标准

### 1. 调用链完整性（1-5 分）

| 分 | 标准 |
|----|------|
| **5** | 列出全部 7 层（server/registry/tool/summarizer/cache/data/llm），各层职责描述准确 |
| **4** | 5-6 层，仅漏其中一层（如忽略 registry），或描述不够精确但大方向正确 |
| **3** | 3-4 层，提到 summarizer 和 llm，但 data/cache 层模糊或合并 |
| **2** | 只到 summarizer 一层，未往下展开 cache/data/llm |
| **1** | 泛泛描述（如"经过若干模块"），无实质层次信息 |

### 2. 调用链深度（实际描述了几层）

| 描述层数 | 情形 |
|---------|------|
| **7** | server / registry / tool / summarizer / cache / data / llm 全部说到 |
| **6** | 缺 1 层（常见：忽略 registry，或把 cache/data 合并为"数据层"计 1 层）|
| **5** | 缺 2 层（常见：cache 和 data 各省一层）|
| **4** | server / tool / summarizer / llm，跳过 registry、cache、data |
| **3** | tool / summarizer / llm |
| **≤2** | 严重缺失 |

### 3. 是否说到 LLM 调用层 ✅ / ⚠️ / ❌

**满分答案**：
- 说明 `summarizer.py` 中调用了 `llm.call_llm()`
- 传入的 prompt 含模块文件列表、符号签名、上下游依赖
- 指出这是**主要性能瓶颈**（网络+推理延迟）

**评判规则**：
| 标志 | 条件 |
|------|------|
| ✅ 完整 | 提到"调用 LLM"且**指出是主要瓶颈** |
| ⚠️ 部分 | 提到"调 LLM"但未说是主瓶颈，或未说明传了什么数据 |
| ❌ 缺失 | 完全未提及 LLM 调用 |

---

### 4. 是否说到 cache 读写层 ✅ / ⚠️ / ❌

**满分答案**：
- 缓存读（入口）：`cache.read_module()` 检查是否已有缓存，命中则直接返回
- 缓存读（索引）：`cache.read_modules_index()` 查模块名→文件列表映射
- 缓存写（末尾）：`cache.write_module()` 将 LLM 摘要写入磁盘
- 默认模式不检查 hash，直接读缓存

**评判规则**：
| 标志 | 条件 |
|------|------|
| ✅ 完整 | 提到缓存命中检查（read）+ 写入两个方向 |
| ⚠️ 部分 | 只提"有缓存"或"会写缓存"，未说明读写两个方向 |
| ❌ 缺失 | 完全未提及 cache |

---

### 5. 是否说到 data 数据查询层 ✅ / ⚠️ / ❌

**满分答案**：
- 在 `summarizer.py:238-254` 打开 `CodeGraphDB`
- 调用 `list_modules(db)`、`module_dependencies(db)`、`directory_dependencies()`
- 调用 `db.iter_nodes(kinds=("function","class","method"))` 获取模块内符号
- 这些都是对 CodeGraph SQLite DB 的查询（即 `.codegraph/codegraph.db`）

**评判规则**：
| 标志 | 条件 |
|------|------|
| ✅ 完整 | 提到从 CodeGraph DB 查询，说明查了哪几类数据（模块列表/依赖边/符号） |
| ⚠️ 部分 | 只说"从数据库查"，未区分查了什么类型 |
| ❌ 缺失 | 完全未提及 data 层 / CodeGraph DB 查询 |

| 分 | 标准 |

| **5** | 识别 LLM 为主瓶颈 + 提到 DB 全量遍历 + 提到 db_hash 计算（条件性）+ 说明缓存命中时无瓶颈 |

| **1** | 瓶颈分析不合理（如认为 JSON 解析是主要瓶颈，或忽略 LLM） |
