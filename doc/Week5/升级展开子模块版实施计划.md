# 升级展开子模块版实施计划

> 目标：在保留现有 `project_map` 资源 / `explore_module` 工具不变的前提下，新增两个 v2 版本，支持模块的子模块展开（递归到 2 层）。
>
> 后续对话按本文档逐节实现，不确定处必须问用户，禁止猜测。
>
> 关联文档：
> - 总执行计划：`doc/vibecoding_rules/codesense-intern-project-plan.md`（Stretch Goal "多粒度 explore"）
> - v1 实现参考：`doc/Week4/LLM模块界定实现.md`
> - 现有项目交接：`doc/Week5/week5_handoff.md`

---

## 0. 关键决策汇总（已与用户确认）

| # | 决策点 | 最终选择 |
|---|-------|---------|
| D1 | 触发时机 | **阈值自动触发**：模块文件数 > 20 → 首次 explore 时生成子模块 |
| D2 | API 形式 | **路径式**：`explore_module_v2("核心库")` 或 `explore_module_v2("核心库/extraction")` |
| D3 | 生成时机 | **Lazy**：`explore_module_v2` 首次调到大模块时才生成子模块（不在 `project_map_v2` 阶段预生成） |
| D4 | project_map_v2 是否展子模块 | **不展示**：保持鸟瞰简洁，子模块仅在 explore_v2 输出中出现 |
| D5 | 深度限制 | **2 层封顶**：顶层 + 子模块，不做第 3 层 |
| D6 | 与 v1 共存 | **完全独立**：v1 文件、API、缓存全部不动；v2 走独立缓存命名 |
| D7 | 子模块生成的输入 | 仅限于父模块的 `directories` 内的文件、符号、目录依赖 |
| D8 | 子模块文件数下限 | 父模块 ≤ 20 文件 → 不拆分；拆分后仍按 LLM 自由划分（不强制分多少个） |

## 2. 缓存结构设计

### 2.1 整体目录布局

```
.codesense/
├── project_map.md                ← v1 不变
├── modules_index.json            ← v1 不变
├── modules/<safe_key>.md         ← v1 不变
├── project_map_v2.md             ← 【新增】v2 概览（Markdown 模板渲染）
├── modules_index_v2.json         ← 【新增】v2 模块树（含 submodules 字段）
├── modules_v2/<safe_key>.md      ← 【新增】v2 模块/子模块摘要（Markdown）
└── meta.json                     ← 共用，db_hash 不变
```

### 2.2 `modules_index_v2.json` Schema

```json
{
  "generated_at": "2026-06-18T...+00:00",
  "modules": [
    {
      "name": "核心库",
      "description": "...",
      "directories": ["src"],
      "files": ["src/index.ts", "src/utils.ts", ...],
      "submodules": [
        {
          "name": "extraction",
          "description": "...",
          "directories": ["src/extraction"],
          "files": ["src/extraction/orchestrator.ts", ...]
        },
        {
          "name": "mcp",
          "description": "...",
          "directories": ["src/mcp"],
          "files": [...]
        }
      ]
    },
    {
      "name": "测试工具集",
      "description": "...",
      "directories": ["__tests__"],
      "files": [...],
      "submodules": []
    }
  ]
}
```

**关键字段说明：**
- `submodules` 字段始终存在；未生成过则为 `[]`
- 子模块本身**不再嵌套** `submodules` 字段（D5 深度封顶）
- 子模块的 `directories` 必须是父模块 `directories` 下的子集

### 2.3 `safe_key` 路径化

对路径式名称做归一化后再 hash：

```python
def safe_key_v2(path: str) -> str:
    """Path-style: 核心库/extraction → sha1(normalized)[:12]"""
    norm = "/".join(p.strip().lower() for p in path.split("/") if p.strip())
    return hashlib.sha1(norm.encode("utf-8")).hexdigest()[:12]
```

> 顶层模块 `核心库` 与子模块 `核心库/extraction` 哈希值不同；空格、大小写归一化保持与 v1 一致。

### 3.1 行为规范

| 步骤 | 内容 |
|------|------|
| 1. 缓存检查 | 读 `meta.json` 的 db_hash → 一致 → 读 `project_map_v2.md` 命中即返回 |

## 4. 新增 Tool：`explore_module_v2`

### 4.1 入参

```python
EXPLORE_MODULE_V2_INPUT_SCHEMA = {
    "type": "object",
    "properties": {
        "module_path": {
            "type": "string",
            "description": "模块名或路径，如 '核心库' 或 '核心库/extraction'。顶层模块名取自 project_map_v2；子模块名由首次调用顶层时 LLM 生成。",
        }
    },
    "required": ["module_path"],
    "additionalProperties": False,
}
```

> 入参字段名沿用 `module_path`（路径式），不要叫 `module_name`，避免和 v1 误解一致。

### 4.2 主流程（伪代码）

```python
async def explore_module_v2(module_path: str) -> str:
    norm = module_path.strip()
    if not norm:
        raise InvalidArgumentError("module_path 不能为空")

    project_root = ...
    cs_dir = project_root / ".codesense"
    db_path = ...
    current_hash = cache.db_hash(db_path)
    mkey = cache.safe_key_v2(norm)

    # 1. 缓存命中
    if cache.is_cache_valid(cs_dir, current_hash):
        cached = cache.read_module_v2(cs_dir, mkey)
        if cached is not None:
            return cached
    else:
        cache.invalidate_v2(cs_dir)

    # 2. 读 modules_index_v2
    index = cache.read_modules_index_v2(cs_dir)
    if index is None:
        raise InvalidArgumentError(
            "请先读取 codesense://project_map_v2 资源生成模块划分"
        )

    parts = [p.strip() for p in norm.split("/") if p.strip()]
    if len(parts) == 1:
        # 顶层模块：可能需要触发子模块生成
        return await _explore_top_module(project_root, cs_dir, index, parts[0], mkey, current_hash)
    elif len(parts) == 2:
        return await _explore_submodule(project_root, cs_dir, index, parts[0], parts[1], mkey, current_hash)
    else:
        raise InvalidArgumentError("不支持深度 > 2 的路径（v2 仅支持顶层/子模块）")
```

### 4.3 顶层模块流程：`_explore_top_module`

```
1. 在 index["modules"] 中按归一化名称查找父模块 entry。
   找不到 → InvalidArgumentError，列出 available 顶层名。

2. 检查 entry["files"] 数量：
   - ≤ 20：走单层流程（同 v1 explore_module 逻辑），直接生成摘要并写缓存返回。
   - > 20 且 entry["submodules"] == []：触发子模块生成（详见 §4.5），更新 index["submodules"] 并写回 modules_index_v2.json。

3. 构建 LLM prompt（_build_module_prompt_v2）：
   - 同 v1 的 _build_module_prompt 内容（含 file_symbols + 上下游依赖）
   - 末尾追加 "## 子模块列表"：列出该模块下已生成的子模块名 + 一句话职责，引导 AI 进一步调用

4. 调 LLM → 写 modules_v2/<safe_key>.md → 返回。
```

### 4.4 子模块流程：`_explore_submodule`

```
1. 在 index 中找 parent entry，再找 submodules 中匹配 child name 的子模块。
   找不到 parent → 列出可用顶层；找不到 child → 列出 parent.submodules 中的可用名。

2. 子模块的 dir_deps、file_symbols 数据准备：
   - dir_deps：用 directory_dependencies 全量数据，但渲染 prompt 时只关心
     "submodule.directories 内 → 任何目标" 和 "任何源 → submodule.directories 内"
   - file_symbols：从 db.iter_nodes 中筛 submodule["files"] 范围
   - inbound/outbound 依赖按"目录"粒度推导（与 v1 _build_module_prompt 同口径）

3. 调 LLM（_build_module_prompt_v2 同模板，传子模块 entry）→ 写缓存 → 返回。
```

### 4.5 子模块生成：`_generate_submodules`

调用时机：顶层模块文件 > 20 且 `submodules == []`。

```
输入：父模块 entry（含 directories + files）

1. 构建子模块划分 prompt：
   - 复用 _build_project_map_prompt 的竖线文本格式
   - 但只喂入"父模块 directories 内的子目录"作为输入
   - directory_symbols 只包含父模块文件
   - directory_dependencies 仅父模块内部依赖（filter both src/tgt 在父 directories 内）

2. _call_llm_for_modules 拿到子模块列表（每条：name, description, directories）

3. _expand_module_files 从父模块 files 中按子目录前缀展开 files

4. 校验子模块的 directories 必须是父模块 directories 的真子集
   - 若 LLM 给出超出父模块范围的目录，过滤掉
   - 若过滤后某子模块 directories 为空，丢弃该子模块

5. 写回 index：parent_entry["submodules"] = 子模块列表

6. 调用 cache.write_modules_index_v2(cs_dir, index["modules"], current_hash)
   - 同步清 modules_v2/<父模块所有相关 key>.md（避免子模块新增后旧缓存陈旧）

返回：填充后的子模块列表（供 _explore_top_module 在主摘要末尾追加）
```

### 5.2 新增函数

| 函数 | 行为 |
|------|------|
| `read_project_map_v2(cs_dir) -> str \| None` | 读 `project_map_v2.md` |
| `write_project_map_v2(cs_dir, content, current_hash)` | 写 `project_map_v2.md` + 更新 meta |
| `read_modules_index_v2(cs_dir) -> dict \| None` | 读 `modules_index_v2.json` |
| `write_modules_index_v2(cs_dir, modules, current_hash)` | 写 `modules_index_v2.json` + 同步清 `modules_v2/`（D7 兼容） + 更新 meta |
| `read_module_v2(cs_dir, mkey) -> str \| None` | 读 `modules_v2/<mkey>.md` |
| `write_module_v2(cs_dir, mkey, summary, current_hash)` | 写 `modules_v2/<mkey>.md` + 更新 meta |
| `safe_key_v2(path) -> str` | 路径式 hash（见 §2.3） |
| `invalidate_v2(cs_dir)` | 仅清 `project_map_v2.md` / `modules_index_v2.json` / `modules_v2/*.md`（**不动 v1 文件**），meta.json 共用 |

### 5.3 `invalidate_v2` 与 v1 共用 meta.json 的处理

`meta.json` 由 v1 的 `invalidate()` 删除，但 v2 调用 `invalidate_v2()` 时 **不删 meta.json**——保留共用。  
v1 的 `invalidate()` 改不改？**不改**。v1 失效会顺带删 meta，导致 v2 缓存被判定无效（is_cache_valid 返回 False），下次 v2 调用会重生 → 安全。

### 5.4 `__init__.py` 导出

把所有新增 v2 函数加入 `cache/__init__.py` 的 `__all__`。

## 9. 测试要求

### 9.1 新增测试文件

| 文件 | 覆盖内容 |
|------|---------|
| `tests/test_cache_v2.py` | `safe_key_v2` / `read/write_modules_index_v2` / `invalidate_v2` 不影响 v1 |
| `tests/test_summarizer_v2.py` | `project_map_v2_summary` / `module_summary_v2` 各分支（缓存命中、首次生成、子模块触发、深度限制） |
| `tests/test_explore_module_v2.py` | Tool 入参校验、路径解析、错误消息 |
| `tests/test_resources_project_map_v2.py` | Resource 读取与 URI 注册 |

### 9.2 关键用例

- **顶层小模块（≤20）**：不触发子模块生成
- **顶层大模块（>20）首次调用**：触发子模块生成 + 索引被更新
- **顶层大模块第二次调用**：命中缓存，不再调 LLM
- **子模块路径合法**：返回子模块摘要
- **子模块路径不存在**：错误消息列出 parent 下可用子模块名
- **3 层路径**：报错 "不支持深度 > 2"
- **v2 失效不影响 v1**：调用 `invalidate_v2` 后 v1 文件仍在
- **modules_index_v2 写入清 modules_v2 子缓存**：旧子模块缓存不残留

### 9.3 测试约束

- LLM 调用全部 mock
- `mypy --strict` 零错误
- `ruff check` 零警告
- 总测试数 ≥ 当前 142 + 新增

## 12. 风险与开放问题

### 12.1 已知风险

| 风险 | 应对 |
|------|------|
| LLM 子模块划分超出父模块目录 | `_filter_subdirs` 过滤 + 校验 |
| 子模块过细（每个 1-2 文件） | LLM prompt 约束 "子模块至少 5 个文件"，超细的合并到 misc |
| LLM 输出空子模块列表（认为不需要拆） | 接受空结果，标记 submodules=[] 但下次仍会重试（缓存方式：只有当 submodules 非空才写回？需用户确认） |
| 父模块 directories 跨多目录（如 ["src", "lib"]）的子模块归属 | 各自分别拆分还是统一拆？建议：统一作为输入喂给 LLM，让 LLM 自己决定 |

### 12.2 实现者必须问用户的开放问题

实现时如下情况仍不确定，**禁止猜测**，必须问用户：

1. 文件数阈值（D1 给的 20）是否需要做成配置项？
2. LLM 给出空子模块时（认为不需要拆），是否每次重试还是缓存为"已拆但无子模块"？
3. 父模块跨多 directories 时，子模块划分是否分别拆还是统一拆？
4. 子模块的"内部接口"提取范围与 v1 一致吗（仍排除 `_` 开头）？
