# Week 5 实验设计：证明/证伪 CodeSense 的实际效果

> 目的：通过受控对比实验，定量评估 CodeSense（MCP Server + Skill）在 AI 辅助编程任务中的实际效果——是否真的让 Agent 更好地理解代码架构，是否反而导致"假饱和"等副作用。
>
> 设计参考：网易内部文章《让 Agent 读 Code Graph，方案设计的反而更差了？》——分层检索 + "假饱和"警惕 + 多维量化指标。

---

## 1. 实验目标

### 1.1 核心命题（需证明/证伪）

| 命题 | 期望结果 |
|------|---------|
| **H1**：CodeSense 提高了 Agent 对跨模块改动的"找全率" | 组3/组4 找全率 > 组2 > 组1 |
| **H2**：CodeSense 没有显著增加误改 | 组3/组4 误改率 ≤ 组2 ≤ 组1 |
| **H3**：CodeSense 没有引发"假饱和"——Agent 读完 project_map 后仍保持深入探索 | 组3/组4 的 阅读数 / 改动数 比值不显著低于组2 |
| **H4**：Skill 对 explore_module 调用有边际增益 | 组4 explore_module 调用率 > 组3 |
| **H5**：CodeSense 没有显著增加 token 消耗 | 组3/组4 token 同组2 持平或更低 |

### 1.2 次要观察项

- LLM 模块划分是否被 Agent 正确理解和使用
- codegraph-main（TypeScript，`src` 被合并为"核心库"）上 explore_module 的有效性
- Agent 在不同任务类型下对 CodeSense 工具的偏好差异

## 3. 任务清单（定稿）

### 3.1 设计原则

- **不在 Phase 1 提示词中明示**"用 CodeSense / project_map / explore_module"，靠任务措辞自然触发工具调用（测真实自然触发能力，避免污染）
- **任务复杂度来自抽象度**而非仓库规模——重构 / 跨模块流程理解 / 跨子系统功能评估
- **改代码任务仅 1 个**（任务1）控制实验执行成本；其余 3 个为纯问答或评估，无需重置代码

### 3.2 任务分类

| 类型 | 量化能力 | 主要指标 |
|------|---------|---------|
| **跨模块改造**（任务 1） | 有 ground truth | 找全率、误改率、测试通过 |
| **跨模块流程分析问答**（任务 2） | 有概念清单 | 概念命中率、幻觉数、调用链深度 |
| **跨子系统协作问答**（任务 3） | 有概念清单 | 概念命中率、幻觉数 |
| **跨子系统功能新增评估**（任务 4） | 有 ground truth | 找全率、是否找到注册模式 |

### 3.3 4 个任务概要

> 完整内容（含 Phase 1/2 提示词、Ground Truth、结果表）见 `doc/Week5/实验过程/任务1_*.md` 到 `任务4_*.md`。

| # | 任务 | 项目 | 类型 | GT | 次数 |
|---|------|------|------|----|------|
| **任务 1** | cache 模块改为"按 hash 增量失效" | CodeSense_V1 | 改代码 | 3 核心 + 2 条件 | 3×4=12 |
| **任务 2** | 分析 explore_module 调用流程和性能瓶颈 | CodeSense_V1 | 问答 | 12 概念 | 3×4=12 |
| **任务 3** | 分析 codegraph_explore 涉及的子系统协作 | codegraph-main | 问答 | 8 概念 | 3×4=12 |
| **任务 4** | 评估 codegraph 加 Erlang 语言支持的改动点 | codegraph-main | 评估清单 | 4 核心文件 + 2 加分 | 3×4=12 |

**合计 48 次实验**，预估 8-12 小时。

### 3.4 任务设计要点

- **任务 1**：跨 3 核心文件（+2 条件），含 `summarizer` 对 cache 的隐含依赖——反假饱和关键测点
- **任务 2**：6 模块调用链（server→registry→tools→summarizer→cache/data/llm）——测 Agent 是否深入到底层
- **任务 3**：codegraph 的 MCP→图查询→DB 流程——测陌生 TS 项目上的架构理解
- **任务 4**：Agent 必须从 python.ts/go.ts 学到注册模式——测 CodeSense 在大型项目"找模式"场景的价值

## 4. 评估指标体系

### 4.1 过程指标（任何任务都测）

从 CodeMaker UI 直接读取，每次实验后立刻填表（关闭对话后数据会丢失）。

| 指标 | 取值方式 | 含义 |
|------|---------|------|
| **检索数** | `grep_search` + `glob_search` + `list_files_*` 调用次数 | Agent 搜索行为 |
| **阅读数** | `read_file` 调用次数（含重复读同文件） | Agent 阅读深度 |
| **改动数** | `edit` + `write` 修改的**去重文件数** | 实际改动 |
| **CodeSense 工具用量** | `access_mcp_resource(project_map)` + `use_mcp_tool(explore_module)` 次数 | CodeSense 触达 |
| **CodeGraph 工具用量** | `codegraph_*` 系列调用次数 | CodeGraph 触达 |
| **总工具调用数** | 全部 tool calls | 整体活动 |
| **Token 消耗** | CodeMaker 显示的总 token | 成本 |
| **耗时** | wall clock（秒） | 效率 |

### 4.2 派生指标（反"假饱和"重点）

| 指标 | 公式 | 解读 |
|------|------|------|
| **阅读/改动比** | 阅读数 / 改动数 | 越高 = 看了不改（可能假饱和或迷茫）；越低 = 改了没看（盲目修改） |
| **检索/阅读比** | 检索数 / 阅读数 | 越高 = 搜了不读（盲目搜索） |
| **CodeSense 命中率** | (project_map + explore_module 调用次数) / 总工具调用数 | CodeSense 实际占用 Agent 注意力的比例 |

### 4.3 结果指标（有 ground truth 的任务才测）

| 指标 | 公式 |
|------|------|
| **找全率（Recall）** | Agent 改/列出的 ground truth 文件数 / ground truth 总数 |
| **误改率（False positive rate）** | Agent 改的非 ground truth 文件数 / Agent 总改动数 |
| **测试通过** | 0 / 1（pytest 是否全过） |
| **概念命中率** | Agent 提到的 ground truth 概念数 / ground truth 概念总数（仅问答类） |
| **幻觉数** | Agent 提到但实际不存在的模块/概念/文件数 |

### 4.4 综合质量分（人工评估）

每个任务结束后，对 Agent 的最终交付物（修改的代码 / 问答答案）打分：

| 维度 | 1分 | 3分 | 5分 |
|------|-----|-----|-----|
| 改动正确性 | 跑不通 / 大面积错 | 主要功能对，细节有问题 | 完全正确 |
| 代码风格一致性 | 完全不一致 | 部分匹配 | 与现有代码一致 |
| 架构理解 | 完全没看模块边界 | 部分理解 | 体现出全局视野 |

## 5.5 任务提示词模板（两阶段提问）

> **关键原则**：Ground Truth 不能在 Phase 1 给出，否则 Agent 会直接照着 GT 修改，污染过程指标（找全率会失真为 100%，无法测出 CodeSense 的真实贡献）。
>
> 两阶段在**同一个 CodeMaker 会话**中执行：Phase 1 跑任务，Phase 2 追问要求对照评估。Agent 在同一会话保留 Phase 1 的完整工具调用记录，能准确算出找全率 / 误改率。

### Phase 1 — 只发任务（不给 GT）

```
## 任务

[粘贴任务描述，例如：把 cache.safe_key() 函数重命名为 cache.module_key_hash()，所有调用点同步修改，测试仍全部通过]

## 验收命令

[uv run pytest -q / codegraph langs / 等]

## 要求

完成任务后，按照全局规则（`.codemaker/rules/experiment-reporting.mdc`）输出完整实验报告。
```

执行后 Agent 输出**第一份报告**（含过程指标、改动文件清单、工具调用序列、假饱和自查、测试结果、质量自评）。

→ 你立刻从 CodeMaker UI 抄 **Token + 耗时**，并把第一份报告内容存档。

### Phase 2 — 在同一会话追加 GT 对照

```
## Ground Truth 对照

本任务的标准答案文件清单如下：

必改文件：
1. [文件路径1]
2. [文件路径2]
（从实验设计.md §3.2 对应任务的 Ground Truth 范围复制）

不该改的范围：上述清单以外的所有源文件

请基于你在 Phase 1 实际执行的改动，对照计算并输出：

| 指标 | 数值 |
|------|------|
| GT 必改文件总数 | ? |
| 实际改动文件中命中 GT 的数 | ? |
| **找全率** = 命中数 / GT 总数 | ? |
| 实际改动文件中**不在** GT 内的数 | ? |
| **误改率** = 不在 GT 内的改动数 / 实际改动总数 | ? |
| 漏改的 GT 文件清单 | （逐行列出，无则填"无"） |
| 误改的非 GT 文件清单 | （逐行列出，无则填"无"） |
```

→ 把第二份报告的结果指标补充到记录表。

### 任务 D（架构问答类）的特殊处理

任务 D 没有"改动文件"概念，两阶段调整为：
- **Phase 1**：只问"用 200 字描述 codegraph 怎么把 AST 转 SQLite，列出涉及子系统"
- **Phase 2**：给出关键词清单 + 子系统清单，要求 Agent 自查"概念命中率 / 幻觉数"

## 9. 风险与开放问题

### 9.1 已知风险

| 风险 | 应对 |
|------|------|
| codegraph-main 上 `src` 被 LLM 合并为单一"核心库"，导致 explore_module 粒度过粗 | 实验记录此局限；任务 D / E 选择不依赖细分子系统的提问 |
| LLM 随机性导致同任务多次结果差异大 | 关键任务（A, D）跑 3 次取中位 |
| Agent 在 4 组之间记忆/缓存污染 | 每次实验都新开一个 CodeMaker 对话 |
| CodeMaker 工具计数粒度不一致 | 以 CodeMaker UI 末尾显示的总数为准；单次实验完成后立刻截图 |
| 任务难度过低，4 组均轻松完成无区分度 | 任务设计阶段刻意挑跨 3+ 模块的任务 |

### 9.2 已知开放问题

1. **是否额外加任务**：建议补充 1 个"明显不该用 CodeSense"的对照任务（如"修改 README 拼写错误"），观察 Agent 是否还会过度调用 explore_module
2. **综合质量分由谁评**：用户自己评 vs 让另一个独立 LLM 评（rubric-based）
3. **数据汇总后是否做统计显著性检验**：样本量小（3 次或 1 次），可能只能做描述性统计
