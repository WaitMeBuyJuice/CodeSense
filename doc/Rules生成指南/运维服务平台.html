Rules（规范）

什么是Rules？

Rules为仓库智聊提供系统级、持久化的指令，能够为你的项目提供关键的上下文、编码偏好或特定工作流，确保 AI 在生成代码、解答问题或辅助开发时，始终遵循统一的标准和背景信息。

仓库智聊支持两种核心类型的Rules：

  • 团队Rules (Team Rules)
    在 DevCloud 研发知识空间中统一管理的团队级Rules，与“项目知识集”绑定，在团队范围内共享。

  • 本地Rules (Local Rules)
    存放于项目仓库的 .codemaker/rules 目录下，支持配置生效Rules。

Rules的工作方式

大型语言模型在多次对话之间不保留记忆。Rules通过在每次请求的提示词（Prompt）开头注入预设的上下文，来解决这个问题。

启用后，Rules内容会成为模型上下文的一部分，为仓库智聊提供一致的指导，就像一位时刻了解项目背景和团队规范的伙伴。

在仓库智聊的上下文中，对话生效Rules都会清晰地显示在对话上方，让你随时了解哪些上下文正在生效。

本地Rules (Local Rules)

本地Rules为你当前的项目代码库提供具体、详细的上下文。它们存放于项目根目录下的 .codemaker/rules 文件夹中。

使用本地Rules可以：

  • 固化与你代码库相关的领域知识（例如：特定 API 的用法）。
  • 自动化项目特定的流程或模板（例如：创建新组件的样板代码）。
  • 规范化代码风格或架构决策（例如：React Hooks 的使用规范）。

Rules结构

每个本地Rules都是一个以 .mdc 结尾的文件，它包含 元数据 (Frontmatter) 和 Rules内容 两部分。

---
description: 请填写Rules描述  // Rules的简短描述，可选
alwaysApply: true  // 控制Rules的生效方式
---
<rule>
Rules内容写在这里
</rule>

  • 元数据 (Frontmatter)：位于文件顶部的 --- 块内，用于配置Rules的行为。

  • Rules内容：位于元数据块之后，可以编写具体的指令、示例代码或引用文件。

创建与管理本地Rules

在 Rules 面板点击【管理Rules】按钮，可进入本地 Rules 配置页面。

  • 点击 “+ 新建Rules” 在 .codemaker/rules/ 目录下创建一个新的Rules文件。
  • 在列表中可以查看所有Rules、它们的生效方式，并进行编辑或删除。
  • 生效Rules：
    • 始终生效：任何情况下都生效
    • 手动指定：手动@引用才生效
    • 指定文件：当引用某个文件时该Rules才生效，需填写文件名匹配格式

生效方式

可以在配置页面或直接修改文件中的 alwaysApply 和 globs 属性来控制Rules的生效方式。

CodeMaker 提供三种生效方式：

生效方式 描述 配置方式

始终生效

Rules将始终被包含在模型上下文中。适用于全局性的、高优先级的指令。

在配置页面选择“始终生效”，或在文件中设置 alwaysApply: true。

手动指定

Rules仅在使用 @Rules文件名 明确提及时才会被包含。适用于工具类或模板类Rules。

在选择“手配置页面动指定”，或在文件中设置 alwaysApply: false 且不提供 globs。

指定文件

当智聊上下文中引用了与某个 glob 模式匹配的文件时，Rules会自动生效。适用于特定模块或文件类型的规范。

在配置页面选择“指定文件”，或在文件中设置 alwaysApply: false 并添加 globs 属性。

指定文件 示例：

假设你有一个Rules用于规范所有 *.service.ts 文件的 RPC 服务定义。

.codemaker/rules/rpc_service_spec.md.mdc

---
description: RPC 服务样板
alwaysApply: false
globs:
  - "backend/services/**/*.service.ts"
---
<rule>
# RPC 服务定义规范
- 定义服务时必须使用我们内部的 RPC 模式。
- 服务名称始终使用 snake_case 命名。
- 必须包含健康检查端点。
@service-template.ts
</rule>

当你在智聊中 @ 任何路径匹配 backend/services/**/*.service.ts 的文件时，这条Rules就会被自动应用。

兼容cursor rules

如果仓库之前使用过cursor等AI编码工具并创建过rules，可以在仓库智聊中看到与使用对应的rules。

如cursor rules在仓库智聊中标记如下， .cursor/rules/ 目录下不变，兼容原先的生效方式（原先为 Apply Intelligently （智能生效）的会自动调整为手动生效）。

团队Rules (Team Rules)

团队Rules提供了一种在整个团队或组织内强制执行编码标准、安全规范和架构决策的强大方式。管理员可以在云端统一配置，团队成员打开对应仓库即可自动获取这些Rules。

管理团队Rules

在 DevCloud 的 【项目知识集】 页面创建和管理团队Rules

  1. 进入项目的“项目知识集”配置页面，新建/编辑知识集。
  2. 在 “Team Rules (团队编程Rules)” 部分，可以添加新的Rules。
  3. 每条Rules包含 “Rules名称” 和 “Rules详情”。

格式与应用方式

  • 格式：团队Rules为纯文本格式，直接在“Rules详情”中填写即可，无需任何特殊的元数据或语法。

  • 适用范围：【项目知识集】支持定义仓库路径，当项目同学打开对应的仓库路径或手动选择该知识集后，其中定义的团队 Rules 将自动生效。

  • 标识：在 Rules 管理面板中，会标识 Rules 来源，方便识别。

最佳实践

为了让Rules发挥最大效用，请遵循以下建议：

  • 聚焦、可执行：Rules应该是清晰、无歧义的指令，而不是模糊的描述。
  • 拆分大型Rules：将复杂的Rules拆分成多个可组合的小Rules，通过“手动指定”或“指定文件”的方式按需组合。
  • 提供示例：在Rules内容中提供具体的代码示例，能让 AI 更好地理解你的意图。
  • 善用团队Rules与本地Rules的组合：
    • 团队Rules：用于定义全局、强制性的标准，如安全规范、团队级代码风格。
    • 本地Rules：用于定义项目特有的上下文，如模块 API、组件模板、特定领域的业务逻辑。

通过有效利用这两种Rules，可以仓库智聊打造成一个真正理解团队的智能编程助手。
